UML Checking

Checking Implementation of UML Sequence Diagrams

 

This is a tool that checks if a Java program correctly implements a set of UML 2.x sequence diagrams. To use the tool, you need to prepare a set of UML sequence diagrams drawn by Altova UModel and exported to XMI format and the Java program that implements the sequence diagrams. Then follow the below steps.

Import our project to Eclipse.

For example: We have a xmi file named widthraw.xmi placed at subfoler xmi, and package withdraw in subfolder src/test  as below figure:

jpf-uml

   |--xmi

        |--withdraw.xmi

   |--src

        |--test

             |--withdraw

                   |--ATM.java

                   |--User.java

                   |--Account.java

 

Annotate the methods that implements sequence diagrams using our annotation. A method can has several @UmlSpec() annotations.

@UmlSpec(sequenceDiagram = { "path to XMI file" }) //annotation

 

For example: In ATM.java we have:

@UmlSpec(sequenceDiagram = { ".\\xmi\\withdraw.xmi" }) //annotation

public void withdrawMoney(int amount) //method under test

Run configuration

1.  From Eclipse menu, choose Run -> Run Configurations... then choose a new Java Application. Then in the Main tab: set Main Class to: RunProtocolChecking

 

2.  In Arguments tab: fill your class name that contains the main method. For the above example, it is: withdraw.ATM

*Note:  to see verbose results, we add –v argument:   withdraw.ATM –v

In the first error, the test case is (1000000,1,-1000000), the SUT turn into the path k > 0. In this path, we forgot to implement the message Dialler.transfer(). In runtime, we see that the method CellularRadio.end() is invoked instead of Dialler.transfer(), so we report an error here.

 

In the second error,the test case is (-1000000,-1000000,-1000000), the SUT turn into the path k < 0. In this path, we forgot to implement the message Dialler.end(). In runtime, we see nothing violated here, but the state machine is not in the last state, so we report a “Not yet finished” error.

Above is the transition in the SD that has never been visited in the SUT. This means that we forgot to implement this message (or combined fragment) in the SUT or it is not reachable in runtime.

 

In this case, the loop fragment in the SD implies that the use can withdraw many times. In the SUT, we’ve just withdraw only one times.  To remove this “Coverage” message, we can modify our method under test as below:

@UmlSpec(sequenceDiagram = { ".\\xmi\\withdraw.xmi" })       //annotation

                public void withdrawMoney(int amount){           //method under test

                                if (amount >= 0){

                                                String log = user.withdrawMoney(amount);

                                                System.out.println(log);

                                }

                                String log2 = user.withdrawMoney(amount);

                }
